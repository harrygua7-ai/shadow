# Phase 3 用户角色动画与移动工作流

## 1. 目标

本工作流将一个角色设计转化为真正可运行的 Godot 用户角色：拥有四方向六帧步行动画、透明背景、稳定锚点、`SpriteFrames` 动画资源、共享移动引擎、碰撞和镜头跟随。

最终交付不是概念图或动作展示图，而是：

```text
选择角色 -> 进入小镇 -> WASD 移动
-> 正确播放四方向动画 -> 正确碰撞 -> 镜头稳定跟随
```

Godot 动画实现以官方 stable 指南为依据：

https://docs.godotengine.org/en/stable/tutorials/2d/2d_sprite_animation.html

## 2. 权限边界

### 允许新增或修改

- `agents/phase-3-user-avatar/`
- `assets/characters/user_avatars/`
- `resources/characters/`
- `scenes/player/`
- `scripts/player/`
- `scenes/character_select/`
- `scripts/character_select/`
- `project.godot` 中与 `move_up`、`move_down`、`move_left`、`move_right` 直接相关的 Input Map 配置
- 世界场景中的用户出生点、`UserAvatar` 实例和镜头目标连接

### 只读，禁止修改

- TileMapLayer 数据
- TileSet 资源
- `WorldCollision`
- 既有 `CollisionShape2D`
- 建筑、道路、水域、围墙和地图装饰
- 地点标记、地图尺寸和 Phase 0-2 资产
- 与角色移动无关的场景节点和项目配置

如果角色无法通过某个区域，工作流只能先调整玩家自身碰撞体。怀疑地图碰撞错误时，停止并报告，不得删除、移动或缩小地图碰撞。

## 3. 阶段 0：生成真正的动作原图

### 输入

- 角色外观说明或角色参考图
- 2D 像素画风格要求
- 四方向定义
- 六帧步态定义

### 生成要求

生成一张四行六列的动作原图：

| 行 | 输入 | 视觉方向 | 动画 |
| --- | --- | --- | --- |
| 1 | S / 下 | 正面 | `walk_down` |
| 2 | W / 上 | 背面 | `walk_up` |
| 3 | D / 右 | 右侧面 | `walk_right` |
| 4 | A / 左 | 左侧面 | `walk_left` |

每行六帧必须形成完整循环：

1. 左脚向前接触。
2. 左脚承重，右脚离地。
3. 右脚经过身体下方。
4. 右脚向前接触。
5. 右脚承重，左脚离地。
6. 左脚经过身体下方，并自然连接第 1 帧。

阶段 0 原图只是生成输入，不能直接当作 Godot Sprite Sheet。图像模型可能生成非等距的行列、人物偏移或背景色变化，必须经过后续确定性处理。

### Gate 0

- 四个方向正确，左右没有反转。
- 每行六帧腿部姿势有真实变化，不是重复站姿。
- 角色身份、服装和比例基本一致。
- 没有额外人物、文字、阴影或严重肢体错误。

## 4. 阶段 1：去背与人物检测

### 处理

1. 从原始图像边缘采样背景色。
2. 去除背景并执行去色边处理。
3. 统计透明、半透明和不透明像素。
4. 检测四个独立动作行。
5. 在每行检测六个独立人物轮廓。

不能假设生成图能够按照整张图片宽高平均切成 6 x 4。只有在检测证明网格严格等距时，才允许直接等分。

### Gate 1

- 检测结果严格为 4 行，每行 6 个人物。
- 背景区域完全透明，没有跨单元格的低透明残影。
- 人物边缘不存在明显绿色、白色或其他背景色边。
- 任何人物没有被裁断。

## 5. 阶段 2：裁切、锚点对齐与标准帧

每个检测到的人物先按自身 Alpha 边界裁切，再放回统一的透明画布。

阶段三黄金样本采用：

- 标准帧画布：256 x 256，用于保留当前生成质量；进入 Godot 后按地图比例显示。
- 水平锚点：以头部中心或稳定躯干中心对齐到统一 X。
- 垂直锚点：所有帧脚底接触线对齐到统一 Y。
- 文件顺序：每个方向 `01` 到 `06`。

只对齐整个人物的位置，不扭曲或缩放单独帧来伪造一致性。

### 输出

```text
processed/frames/
├── walk_down_01.png ... walk_down_06.png
├── walk_up_01.png ... walk_up_06.png
├── walk_right_01.png ... walk_right_06.png
└── walk_left_01.png ... walk_left_06.png
```

### Gate 2

- 共 24 张 RGBA PNG。
- 所有图片尺寸一致。
- 头部或躯干中心水平误差不超过 1-2 像素。
- 脚底接触线完全一致。
- 每帧只存在一个人物轮廓。
- 循环播放时没有左右闪烁、重影或突然跳位。

## 6. 阶段 3：动画预演

在进入 Godot 前，用 Canvas 或其他轻量预览器验证：

- W 使用 `walk_up` 背面动画。
- S 使用 `walk_down` 正面动画。
- A 使用 `walk_left` 左侧动画。
- D 使用 `walk_right` 右侧动画。
- 四组动画均按 01 到 06 播放并循环。
- 角色向前或向后移动时身体不左右闪动。
- 左右移动时人物朝向和位置变化一致。
- 斜向速度经过归一化。

阶段 3 失败时返回阶段 1 或阶段 2 修复，不进入 Godot。

## 7. 阶段 4：生成 Godot 规范 Sprite Sheet

24 张标准帧是规范源。工作流按固定顺序重新打包为真正等距的 Sprite Sheet：

```text
6 个水平帧 x 4 个垂直帧

第 1 行：walk_down  01-06
第 2 行：walk_up    01-06
第 3 行：walk_right 01-06
第 4 行：walk_left  01-06
```

Sprite Sheet 每格尺寸必须完全一致，并在元数据中记录：

- `hframes = 6`
- `vframes = 4`
- 单帧宽高
- 行方向映射
- 帧播放顺序
- 建议 FPS
- 循环设置

## 8. 阶段 5：导入 Godot SpriteFrames

Godot 官方支持把独立图片或单张 Sprite Sheet 交给 `AnimatedSprite2D`。本工作流选择 `AnimatedSprite2D + SpriteFrames` 作为基础角色动画主路径，因为它直接对应离散的角色帧和多组命名动画。

### Godot 操作

1. 在 `UserAvatar` 中使用 `AnimatedSprite2D`。
2. 为其创建独立的 `SpriteFrames` 资源。
3. 选择 “Add frames from a Sprite Sheet”。
4. 设置 6 个水平帧和 4 个垂直帧。
5. 分别选择每一行正确的六帧。
6. 创建并命名 `walk_down`、`walk_up`、`walk_right`、`walk_left`。
7. 四个行走动画开启 Loop。
8. 初始 Speed (FPS) 设置为 10。
9. 创建四个 `idle_*` 动画，每个至少一个中立站姿。
10. 使用 SpriteFrames 编辑器的 Play 按钮逐个预览。

`AnimationPlayer` 在阶段三不作为基础行走方案。它更复杂，但能够驱动位置、缩放等额外属性；只有未来出现转身、受击、表情或复合效果时再评估使用。角色物理位置始终由 `CharacterBody2D` 决定。

### Gate 5

- `SpriteFrames` 中存在八个规定名称的动画。
- `walk_*` 每组恰好六帧、顺序正确、循环开启。
- `idle_*` 使用正确朝向的中立站姿。
- 编辑器预览中没有方向错误、跳帧或循环断点。
- 纹理过滤和压缩设置适合像素角色。

## 9. 阶段 6：运行时动画控制

运行时代码根据最终移动方向选择动画：

```text
velocity.y > 0 -> walk_down
velocity.y < 0 -> walk_up
velocity.x > 0 -> walk_right
velocity.x < 0 -> walk_left
velocity == 0 -> 对应最后朝向的 idle_*
```

使用 `AnimatedSprite2D.play(animation_name)` 切换动画。只有目标动画名称发生变化时才调用 `play()`，否则每个物理帧重新播放会让动画不断回到第一帧。

10 FPS 是初始基线。正式手感测试时可以调整 `SpriteFrames` 的 FPS 或 `AnimatedSprite2D.speed_scale`，但动画不能直接改变角色坐标。

## 10. 阶段 7：移动、碰撞与镜头集成

1. `CharacterBody2D` 在 `_physics_process(delta)` 中计算速度。
2. Input Map 读取四个抽象移动动作。
3. 对角输入归一化。
4. 使用 `move_and_slide()` 处理玩家与既有世界碰撞。
5. 动画读取最终运动状态。
6. Camera Rig 跟随玩家位置并受世界边界约束。

地图 Collision 全程只读。角色无法通过时只能调整玩家自己的碰撞形状，或提交地图问题报告。

## 11. 阶段 8：最终验收与变更审查

### 动画验收

- 四方向均完成至少两个完整六帧循环。
- 正面和背面行走没有水平闪动。
- 左右方向正确，没有 A/D 反转。
- 每一幕只有一个清楚人物，不存在背景残影。
- 停止后保持最后方向的待机姿势。

### 工程验收

- 角色能够在测试场景和正式小镇运行。
- 移动、动画、碰撞和镜头使用共享实现。
- Godot 输出没有资源缺失或持续错误。
- TileMap、TileSet、`WorldCollision` 和既有地图节点没有变化。

### 报告

每次运行输出：

```text
输入角色与来源
生成与去背结果
检测到的行列和帧数
锚点误差统计
Canvas 预演结果
SpriteFrames 动画清单
Godot 运行测试结果
新增和修改文件
受保护文件变化检查
已知问题与最终结论
```
