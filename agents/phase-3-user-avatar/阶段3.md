# Shadow 阶段三：用户角色表现与移动系统

## 1. 文档地位

本文档是 Shadow 阶段三的目标、范围、工程架构和验收标准。阶段三内的设计、素材、Godot 场景、脚本、测试和专项 Agent 任务都必须服从本文档，并继续遵守仓库根目录的 `DMD.md` 与 `AGENT_PRINCIPLES.md`。

当前前提：Phase 0-2 已完成，小镇地图、TileSet、道路、地点布局和世界碰撞是阶段三的既有输入，不在本阶段重新设计。

阶段三主 Agent 负责拆解任务、确定接口、整合实现和最终验收。专项 Agent 可以研究、制作、实现或审查，但不能擅自改变本阶段目标和边界。

## 2. 阶段三使命

阶段三要完成 Shadow MVP 的第一个“身体感”闭环：用户进入小镇后，拥有一个可识别的角色形象，可以通过输入自然移动；角色朝向、步行动画与实际运动一致；镜头稳定地跟随角色；角色不能穿过地图中的阻挡区域。

本阶段解决的是：

> 用户如何以一个有形、可控、视觉连贯的角色存在于 Shadow 世界中。

这不是单纯实现一个移动脚本，也不是只生成一张人物图片。阶段三必须打通以下完整链路：

```text
角色视觉定义
-> 角色图像生成
-> 去除背景与图像清理
-> 动画帧生成、切分和标准化
-> Godot 资源导入
-> 角色动画状态驱动
-> 用户输入与移动
-> 世界碰撞
-> 镜头跟随
-> 可玩验收
```

## 3. 阶段目标

### 3.1 必须完成的目标

1. 建立一条经过实际验证的角色资产生产线：从角色设计、四方向图像生成、透明背景处理、帧对齐和 SpriteSheet 打包，一直到 Godot `SpriteFrames` 导入。生产第二个角色时不得重新猜测尺寸、帧数、命名、目录、脚底锚点或导入方式。
2. 先完成一个黄金角色，用它冻结视觉比例和技术规格；黄金角色通过后，建立 30-50 个可供用户选择的角色目录。每个正式可选角色都必须拥有角色 ID、显示名称、预览图和完整可移动动画资源，不能只是静态头像。
3. 建立角色选择流程。用户打开游戏后先看见 30-50 个候选角色，通过预览列表选择一个角色；系统保存本次选择，并将对应角色配置交给世界加载流程。未完成选择时不能以随机或写死角色直接进入正式世界。
4. 在 Godot 中建立一个可复用的 `UserAvatar` 角色场景和一套共享移动引擎。30-50 个角色只替换外观、动画和必要的视觉偏移，不复制 30-50 份控制脚本、碰撞逻辑或镜头逻辑。
5. 用户进入小镇时，世界根据选择结果实例化对应角色形象，并把该形象绑定到共享移动引擎。用户随后通过方向键或 WASD 间接控制自己选择的角色，在世界中进行移动、交互和游玩。
6. 支持键盘方向键与 WASD 的八方向移动输入、四方向视觉表现；实现稳定、帧率无关、带碰撞的角色移动。
7. 实现待机、步行动画及正确的朝向保持，确保所有正式可选角色遵守相同动画命名与播放规则。
8. 实现平滑、稳定且受地图边界约束的镜头跟随。镜头跟随当前被用户选中并实例化的角色，而不是绑定某个固定角色资源。
9. 将选定的用户角色放入现有小镇的合法出生点，形成“选择角色 -> 加载角色 -> 进入小镇 -> 移动游玩”的完整可运行流程。
10. 建立必要的调试方式、验收清单、角色清单和资产来源记录，并验证切换不同角色不会破坏移动、动画、碰撞或镜头行为。

### 3.2 体验目标

- 用户进入场景后立即看见自己的角色，而不是占位圆点或无身体镜头。
- 用户在进入世界前可以清楚预览并选择自己想扮演的角色。
- 进入世界后的角色必须与用户刚刚选择的预览角色一致。
- 按下移动键后，角色应快速响应，停止后不漂移。
- 动画速度、脚步节奏和实际位移在视觉上协调。
- 斜向移动不能比水平或垂直移动更快。
- 面向方向不应在低速或同时按键时频繁抖动。
- 镜头跟随应柔和但不迟钝，不应产生明显抖动或越出世界有效区域。
- 角色应正确受到建筑、围墙、水域和其他既有碰撞体限制。

## 4. 阶段边界

### 4.1 本阶段包含

- 用户角色的视觉方向与技术规格。
- 30-50 个角色的可选择目录、预览信息和角色配置。
- 进入世界前的基础角色选择界面与选择状态传递。
- 角色基础形象的生成或选型。
- PNG 去背、边缘清理、像素对齐、尺寸统一。
- 待机与行走动画帧的生成、整理和 SpriteSheet 输出。
- Godot 图像导入配置与 `SpriteFrames` 资源。
- 用户输入映射、速度计算、碰撞移动和出生点。
- 动画状态选择、朝向记忆和播放速率。
- `Camera2D` 跟随、平滑与边界。
- 单机本地运行所需的调试和测试。
- 为未来换装、联网同步和 Agent 角色复用预留稳定接口。

- AI 居民或用户自有 Agent 的自主移动、寻路、决策和行为树。

- 用户自由输入提示词并实时生成完整动画角色。
- 用户在阶段三中自行上传、编辑或生成新角色。
- 完整捏脸、服装商城、分层换装和骨骼动画编辑器。
- 战斗、跳跃、冲刺、体力、载具、推拉物体或复杂物理。
- 地点交互、对话、任务、背包、创建系统和世界记忆。
- 建筑内部切换、传送门和正式场景加载系统。
- 重做 Phase 0-2 的地图布局、TileSet 美术或地点定义。
- 为所有未来角色一次性建立通用实体框架。

如果实现上述排除项对本阶段没有直接阻塞，应记录为后续建议，不进入阶段三代码。

## 5. 核心工程原则

### 5.1 用户层优先

本阶段只交付用户控制角色。Agent 角色以后可以复用视觉表现、碰撞和基础运动能力，但不能为了未知的 Agent 行为提前引入寻路、网络权威或复杂实体组件系统。

### 5.2 离线资产与运行时分离

图像生成、去背、裁切和 SpriteSheet 打包属于离线资产流水线。Godot 运行时只消费已经验证的透明 PNG、元数据和 `SpriteFrames`，不在游戏运行时调用图像模型或执行去背。

### 5.3 运动事实唯一

`CharacterBody2D` 的实际速度是运动状态的唯一事实来源。动画与镜头读取角色状态，不各自重新推断输入，避免出现“角色被阻挡但仍按输入方向错误播放”的多套状态。

### 5.4 输入意图与移动执行分离

输入模块只表达玩家想往哪里走；移动模块负责速度、归一化和碰撞；动画模块负责把最终运动结果映射为视觉状态。未来 AI 或网络控制可以替换输入来源，而无需重写角色表现。

### 5.5 先建立一个黄金样本

阶段三先完成一个质量合格的标准用户角色，并用它验证全链路。只有黄金样本通过验收后，才按照同一规格生产角色目录中的其他角色，最终形成 30-50 个正式候选角色，防止错误规格扩散到所有资产。

### 5.6 角色选择与角色控制分离

用户选择的是“要扮演的角色身份和外观”，不是一套独立的移动程序。每个角色通过数据配置声明自己的 ID、名称、预览图、动画资源和视觉偏移；一个共享的 `UserAvatar` 场景负责输入、移动、碰撞和动画驱动。

因此增加第 31 个角色主要是增加一份合格的美术资产和角色配置，不应复制或修改移动引擎。角色选择结果决定运行时加载哪份配置，键盘输入随后控制该配置呈现出来的角色身体。

## 6. 总体工程架构

阶段三分为两个域：资产生产域和 Godot 运行时域。

```text
资产生产域
角色视觉说明
  -> 原始生成图
  -> 去背与边缘清理
  -> 帧校准与尺寸标准化
  -> SpriteSheet + Metadata
  -> 资产验收
                         |
                         v
Godot 运行时域
Character Catalog -> Character Select -> Selected Character ID
                                            |
                                            v
                                  Character Definition
                                            |
                                            v
Input Map -> Movement Input -> Shared UserAvatar -> CharacterBody2D
                                      |                 |
                                      v                 v
                               Animation Driver       Collision
                                      |
                                      v
                     Selected Character SpriteFrames

CharacterBody2D -> Camera Follow -> Camera2D + World Limits
```

运行时推荐节点结构：

```text
UserAvatar (CharacterBody2D)
├── AnimatedSprite2D
├── CollisionShape2D
├── CameraAnchor (Marker2D)
└── DebugOverlay（仅调试构建，可选）

PlayerCameraRig (Node2D)
└── Camera2D
```

镜头可以作为角色子节点完成最小版本，也可以采用独立 `PlayerCameraRig`。阶段三推荐独立镜头 Rig：角色只暴露跟随目标，镜头系统负责平滑和边界，使镜头职责不会污染角色控制脚本。

在用户进入世界前，`CharacterSelect` 从 `CharacterCatalog` 读取 30-50 个角色配置并展示预览。用户确认后，系统只传递稳定的 `character_id`；世界加载流程再通过该 ID 取得正式动画资源并配置 `UserAvatar`。选择界面不直接创建移动逻辑，移动逻辑也不关心用户是怎样选中该角色的。

## 7. 资产生产架构

### 7.1 角色视觉规范

第一版角色应匹配当前 32 x 32 像素地图网格，并保持成熟、温暖、略带奇异感的 Shadow 风格。建议采用俯视或三分之四俯视的 2D 像素角色。

黄金样本建议规格：

- 单帧画布：48 x 64 px；如果实测与地图比例不匹配，可在黄金样本阶段调整一次。
- 脚底锚点：所有帧保持在相同基准线上。
- 方向：`down`、`up`、`right`、`left`，并与最终 Sprite Sheet 的四行顺序一致。
- 行走帧：每方向 6 帧，顺序为左脚接触、左脚承重、右脚经过、右脚接触、右脚承重、左脚经过，并能从第 6 帧自然循环回第 1 帧。
- 待机帧：每方向至少 1 帧；可以直接使用对应方向的中立站姿。
- 背景：完整 Alpha 透明，不保留白边、色边或阴影底板。
- 轮廓：缩放到游戏实际显示尺寸后仍清楚可辨。
- 光照：各方向一致，不把固定背景光烘焙成不自然的大片阴影。

左右方向允许在风格和服饰不对称要求较低时镜像生成，但导出的动画名仍必须完整，运行时代码不依赖动态翻转。

### 7.2 目录与产物约定

建议角色资产结构：

```text
assets/characters/user_avatars/avatar_001/
├── source/                  # 原始生成图与来源说明
├── processed/               # 去背、清理、对齐后的独立帧
├── avatar_001_spritesheet.png
├── avatar_001_preview.png
└── avatar_001_metadata.json
```

`metadata.json` 至少记录：

- 角色 ID 和版本。
- 单帧宽高、行列数和动画名称。
- 每个动画对应的帧坐标与建议 FPS。
- 脚底锚点或视觉偏移。
- 生成方式、源文件、许可或模型信息。
- 已完成的人工验收状态。

### 7.3 生成与去背流程

角色生成不应要求图像模型一次性完美输出可用 SpriteSheet。可靠流程是先固定角色设计，再生成方向和动作，最后由确定性工具完成去背、裁切和打包。

处理顺序：

1. 用角色说明固定服饰、发型、色彩、比例和辨识特征。
2. 生成中立站姿并确认角色设计。
3. 生成四方向行走关键帧，检查身份一致性。
4. 执行背景移除并人工检查半透明边缘。
5. 对每帧做统一画布、脚底基线、重心和像素网格校准。
6. 去除重复帧、破损肢体、漂移配饰和方向错误。
7. 按固定行列打包 SpriteSheet，并生成元数据和预览图。
8. 在 Godot 的实际地图比例中验收，而不是只看放大图。

任何自动去背结果都必须经过人工视觉验收。头发、衣摆、手持物和脚部是最容易出现白边、缺口与抖动的位置。

### 7.4 Godot 导入规则

- 阶段三基础角色动画统一采用 Godot 官方支持的 `AnimatedSprite2D + SpriteFrames` 路径。`AnimationPlayer` 留给需要同时驱动位置、缩放或其他属性的复杂动画，不用于基础四方向行走。
- 关闭会破坏像素边缘的纹理过滤；具体设置服从项目当前像素渲染配置。
- 不使用有损压缩处理角色主 SpriteSheet。
- 资产流水线先保留 24 张独立透明帧作为可审查的规范源，再确定性打包为 6 列 x 4 行 Sprite Sheet；四行顺序固定为 `down`、`up`、`right`、`left`。
- Godot 导入时创建独立的 `SpriteFrames` 资源，通过 “Add frames from a Sprite Sheet” 按 6 个水平帧、4 个垂直帧切分；不得直接使用图像模型输出的非等距原始动作表。
- 帧区域必须严格对应元数据。导入后逐行动画选择正确的 6 帧，不能把相邻方向或空白帧加入动画。
- `SpriteFrames` 动画统一命名为 `idle_down`、`idle_left`、`idle_right`、`idle_up`、`walk_down`、`walk_left`、`walk_right`、`walk_up`。
- 四组 `walk_*` 动画默认开启循环，初始基线速度设为 10 FPS；最终 FPS 必须结合角色实际移动速度在 Godot 视口和运行场景中校准。
- 四组 `idle_*` 至少包含一个明确的中立站姿，不使用迈腿最明显的接触帧作为默认待机姿势。
- 导入后必须使用 `SpriteFrames` 编辑器的播放按钮逐个预览八个动画，确认方向、帧序、循环衔接、脚底稳定和身体中心稳定后，才允许连接移动代码。
- 资源引用使用稳定路径，不引用 `source/` 中的临时生成图。

## 8. Godot 运行时架构

### 8.1 用户输入模型

阶段三中的“用户输入”不是单一键盘输入，而是一个连续的两阶段过程：

```text
第一阶段：用户选择“我是谁”
打开游戏 -> 浏览 30-50 个角色 -> 查看预览 -> 确认角色
-> 产生 selected_character_id

第二阶段：用户控制“我要去哪里”
进入小镇 -> 系统按 selected_character_id 加载角色外观
-> 用户通过键盘给出移动方向 -> 共享移动引擎移动该角色
```

第一阶段属于角色身份输入。系统必须提供角色目录和选择界面，每个候选项至少显示角色预览图和名称。用户确认后，选择系统只保存稳定的角色 ID，例如 `avatar_017`，不把图片路径或场景节点当成选择结果。世界通过角色 ID 查询角色配置，并把对应 `SpriteFrames`、预览信息和视觉偏移应用到共享的 `UserAvatar`。

第二阶段属于角色操控输入。用户不是直接改变角色坐标，而是通过按键表达移动意图。Godot Input Map 提供以下抽象动作：

- `move_left`
- `move_right`
- `move_up`
- `move_down`

键位至少覆盖 WASD 与方向键。控制脚本只读取动作，不直接检查具体键码，以便未来支持手柄、改键和自动化测试。

移动输入层输出归一化二维方向向量。相反方向同时按下时互相抵消，斜向输入归一化，保证各方向移动速度一致。

这两种输入必须分开保存和处理：`selected_character_id` 决定用户扮演的外观身份，`input_direction` 决定这个角色在当前物理帧想往哪里移动。更换角色外观不得改变移动速度、碰撞规则和镜头行为。

### 8.2 移动引擎

用户角色使用 `CharacterBody2D` 和 `move_and_slide()`。移动计算发生在 `_physics_process(delta)`，显示帧率不能改变角色实际速度。

最小移动数据：

- `move_speed`：可导出调参的每秒像素速度。
- `input_direction`：玩家本帧输入意图。
- `velocity`：角色提交给物理系统的速度。
- `last_facing`：停止时保留的最后有效朝向。
- `is_moving`：由最终速度阈值判断的运动状态。

阶段三默认采用即时加速和停止，以获得清晰响应；如果体验测试证明过硬，可以加入非常短的加减速，但必须保证停止不漂移，并通过导出参数控制。

碰撞形状应围绕角色脚部和躯干下半部，而不是覆盖完整图片。头发和宽松衣服属于视觉区域，不应让角色在墙边产生不合理距离。

### 8.3 动画驱动

动画状态由两个维度组成：

```text
运动状态：idle | walk
朝向状态：down | left | right | up
```

组合后得到八个稳定动画名。朝向计算采用主轴规则：绝对值更大的速度分量决定方向；分量接近时优先保留上一朝向，减少对角移动时的快速切换。

动画驱动必须满足：

- 移动时使用 `AnimatedSprite2D.play(animation_name)`，但只在目标动画名称变化时调用，不能每物理帧重启动画。
- 停止时切换到同方向 `idle`，不默认回到向下。
- 基础行走不使用 `AnimationPlayer` 改变角色位置或缩放，避免视觉动画与 `CharacterBody2D` 的物理位置形成两套运动事实。
- 角色被障碍完全阻挡时，是否继续播放行走应由实际移动结果和体验测试决定，并保持全项目一致。
- 行走动画以 10 FPS 为初始基线，通过 `SpriteFrames` 的 Speed (FPS) 调整；如需按角色速度微调，优先使用 `AnimatedSprite2D.speed_scale`，不复制另一套动画资源。

动画只是表现层，不能直接修改角色物理位置。

### 8.4 镜头系统

镜头系统接收角色的世界坐标作为跟随目标，并独立负责：

- 平滑跟随。
- 视口缩放。
- 地图边界限制。
- 出生时立即对准角色。
- 窗口尺寸变化时维持正确构图。

推荐策略：

- 使用 Godot `Camera2D` 的位置平滑能力或一个简单、可测试的插值 Rig。
- 首帧直接对准目标，避免从世界原点滑向角色。
- 镜头边界从当前地图的有效世界矩形获得，避免硬编码散落在多个脚本。
- 当视口大于某个地图轴时，将镜头居中于该轴，不产生边缘来回跳动。
- 镜头缩放保持角色和附近道路清晰可读，并在常见桌面窗口尺寸下测试。

阶段三的“镜头捕捉”定义为稳定锁定并跟随本地用户角色，不包含电影镜头、对话镜头、目标切换和多人观战。

### 8.5 世界集成层

世界场景负责：

- 接收已经确认的 `selected_character_id`。
- 从角色目录中解析对应的 `CharacterDefinition`。
- 提供用户出生点。
- 实例化共享的 `UserAvatar`，并注入选定角色的动画和视觉配置。
- 提供镜头可用的地图边界。
- 保持既有 `WorldCollision` 有效。
- 确保角色渲染顺序符合建筑和环境遮挡关系。

角色场景不应硬编码小镇出生坐标或地图大小。出生点和地图边界属于世界场景数据，以便角色以后进入其他地区时复用。

## 10. 数据与接口契约

阶段三对外提供以下稳定概念：

- `UserAvatar`：可实例化的本地用户角色场景。
- `CharacterCatalog`：30-50 个可选角色配置的有序目录，是选择界面的唯一数据来源。
- `CharacterDefinition`：单个角色的数据配置，至少包含 `character_id`、显示名称、预览图、`SpriteFrames` 和视觉偏移。
- `selected_character_id`：用户确认选择后产生的稳定 ID，负责跨越选择界面和世界加载流程。
- 移动动作：四个 Input Map action。
- `facing_direction`：角色当前或最后朝向。
- `is_moving`：角色是否正在有效移动。
- `camera_target`：镜头跟随锚点或角色引用。
- `spawn_point`：世界提供的出生位置。
- `world_bounds`：世界提供的镜头有效矩形。

未来 Agent 角色可以复用 SpriteFrames 命名、动画驱动和碰撞标准，但不能直接复用读取用户键盘的控制器。未来联网层应读取并发布角色状态，而不是侵入动画资源结构。

## 11. 实施顺序与阶段门

### Gate 1：资产规范冻结

完成角色比例、帧尺寸、方向、动画名、锚点、SpriteSheet 行列和目录规范。未通过 Gate 1 前不批量生成动画。

### Gate 2：黄金角色资产通过

至少一个角色完成生成、去背、帧校准和 Godot 导入，在实际地图中无白边、无明显抖动、比例合理。

### Gate 3：角色目录与选择流程通过

角色目录达到 30-50 个正式候选角色。选择界面能完整展示目录、确认选择并产生有效的 `selected_character_id`；世界能够加载与预览一致的角色。任何缺少完整动画资源的角色不得进入正式目录。

### Gate 4：移动垂直切片通过

角色可以在测试场景中通过键盘移动、碰撞、切换待机与行走动画，斜向速度正确。

### Gate 5：小镇集成通过

用户选择的角色在现有小镇合法出生，不能穿越核心阻挡，镜头平滑跟随并受到边界限制。

### Gate 6：体验与回归验收

抽样切换多个体型和外观不同的角色，完成不同方向、窗口尺寸、墙角、地图边缘、连续转向和停止启动测试；记录已知限制和 Phase 4 交接事项。

## 12. 验收标准

阶段三只有同时满足以下条件才算完成：

### 资产

- 正式角色目录中存在 30-50 个可选择角色。
- 每个正式角色都有唯一 ID、显示名称、预览图、来源记录和 Godot 角色配置。
- 每个正式角色都拥有透明背景和符合统一规格的帧尺寸。
- 每个正式角色的四个方向均有待机和行走动画资源。
- 各帧脚底基线稳定，无明显画面跳动。
- SpriteSheet、预览图、元数据和来源记录齐全。
- Godot 实际显示尺寸下没有明显白边、破损 Alpha 或严重身份漂移。

### 角色选择

- 游戏进入世界前展示完整角色目录，而不是写死默认角色。
- 用户可以预览、选择并确认一个角色。
- 确认后产生有效且可追踪的 `selected_character_id`。
- 进入小镇后的角色与选择界面中的角色一致。
- 更换候选角色不会复制或改变移动引擎、碰撞和镜头逻辑。
- 无效或缺失的角色 ID 有明确回退行为和错误记录。

### 移动

- WASD 和方向键均可控制。
- 水平、垂直与斜向的最大速度一致。
- 移动与停止响应稳定，帧率变化不改变速度。
- 角色不能穿过现有关键碰撞体。
- 碰撞形状与视觉脚位基本一致。

### 动画

- 四方向切换正确，停止后保持最后朝向。
- 动画不会因每帧重播而停留在首帧。
- 行走节奏与移动速度视觉协调。
- 快速反向、连续转向和对角移动不会产生明显抖动。

### 镜头

- 场景加载首帧正确聚焦用户角色。
- 跟随平滑且没有可见震动或明显延迟。
- 镜头不会显示地图边界之外的无效区域。
- 常见桌面窗口尺寸和项目目标缩放设置下构图有效。

### 集成

- 项目主运行入口可以直接进入可移动的小镇体验。
- 无阻塞性 Godot 错误或缺失资源警告。
- Phase 0-2 地图基线未被无关重做。
- 核心移动逻辑具备最低限度的可重复测试或明确人工测试清单。

## 13. 测试矩阵

至少覆盖以下场景：

| 类别 | 测试内容 | 预期结果 |
| --- | --- | --- |
| 输入 | 单方向、对角、相反方向同时按下 | 方向正确，无速度叠加 |
| 状态 | 启动、停止、快速反向、连续绕圈 | 动画和朝向稳定 |
| 碰撞 | 正面撞墙、沿墙滑动、墙角、窄路 | 不穿透、不异常卡死 |
| 镜头 | 出生、地图中心、四侧边界、角落 | 首帧对准，边界正确 |
| 显示 | 不同窗口尺寸与项目缩放 | 角色清晰，画面无越界 |
| 性能 | 连续移动与动画播放 | 无明显卡顿、错误刷屏 |
| 资产 | 放大检查和游戏比例检查 | Alpha、基线和身份一致 |

## 14. 风险与控制

### 图像生成一致性不足

控制方式：先冻结中立角色设计；分方向生成后人工筛选；先交付并验收一个黄金样本，再按冻结规格扩展到 30-50 个正式角色；保留所有源文件和处理记录。

### 去背造成边缘闪烁

控制方式：逐帧检查 Alpha；在深色、浅色和地图背景上分别预览；统一边缘处理规则。

### 动画帧重心漂移

控制方式：以脚底和身体中心建立校准线；所有帧放在固定画布；在 Godot 中循环播放验收。

### 角色与地图比例不协调

控制方式：在 Gate 1 用单帧站姿放入现有地图实测；比例确认后再生成全套动画。

### 控制器过早泛化

控制方式：只抽象已经存在的变化点，即输入来源、移动参数、动画资源和镜头目标；Agent AI、网络和换装等留到对应阶段。

### 镜头平滑带来输入迟滞感

控制方式：分开测试角色响应与镜头响应；设定合理平滑参数；出生与传送时允许立即捕捉目标。

## 15. 阶段三完成后的交付

阶段结束时应提交：

- 本阶段最终规范及关键决策记录。
- 一个作为质量基准的黄金用户角色，以及总计 30-50 个正式候选角色的源图、处理后帧、SpriteSheet、元数据和 Godot 资源。
- 角色目录、角色选择界面、选择状态保存与世界加载衔接。
- `UserAvatar`、移动控制、动画驱动和镜头 Rig。
- 已集成“选择角色 -> 进入小镇 -> 控制所选角色”的可运行场景流程。
- 自动检查结果或人工验收报告。
- 已知限制和面向 Phase 4 的接口说明。

Phase 4 可以在此基础上加入地点感知、靠近提示和轻量交互。AI 居民和 Agent 角色应在用户角色的移动与表现基线稳定后，再接入各自的控制来源。

## 16. 阶段三主 Agent 的决策原则

当质量、速度和未来扩展发生冲突时，按以下顺序判断：

1. 用户移动与动画是否自然、可玩。
2. 资产流程是否可重复、可追溯。
3. 运行时职责是否清楚、可调试。
4. 是否保护了现有地图基线。
5. 是否为下一阶段保留了稳定接口。
6. 是否值得在 MVP 当前阶段承担额外复杂度。

阶段三的成功标准不是“拥有一套角色技术”，而是用户第一次进入 Shadow 时，确实感觉自己以一个角色存在于这个地方。
