# MAP_STANDARD

## 文件目的

本文档定义 Shadow MVP 外部小镇地图的制作标准。

本标准服务于 Phase 0-2：

- Phase 0：地图标准与概念蓝图
- Phase 1：外部小镇 TileMap
- Phase 2：碰撞规则与碰撞实现

本文档是地图相关 Agent 和后续实现 Agent 的共同依据。

## 地图定位

Shadow 第一张地图是：

> 外部小镇地图。

它代表小镇的外部街道、公共空间、建筑外观和地点入口。

它不是：

- 完整大世界
- 室内地图
- 单一酒馆场景
- UI 菜单
- 纯概念美术图

地图应当作为后续玩家移动、地点进入、NPC 出现、社会现场、昼夜天气和 Agent 系统的空间基础。

## 地图实现方式

最终实现采用：

```text
Godot 2D TileMap
```

地图基于 tile 拼接，而不是一整张不可交互大图。

原因：

- 方便碰撞
- 方便扩展
- 方便后续动态变化
- 方便玩家移动
- 方便地点和事件定位

## 视觉风格

地图风格：

```text
2D 像素风，RPG 斜顶式
```

视觉关键词：

- 成熟
- 温暖
- 市井
- 有公共生活感
- 有一点神秘社交感
- 不幼稚
- 不传统奇幻 RPG
- 不赛博朋克过重
- 不像品牌展厅
- 不像聊天软件背景图

## 概念图标准

Phase 0 概念图用于确认布局和气质，不是最终游戏资产。

概念图应当：

- 展示小镇外部整体结构
- 明确各地点相对位置
- 展示道路、广场、小巷、建筑外观
- 建筑不要过大
- 街道和广场要足够宽
- 树和障碍物不要过多
- 留出玩家移动空间
- 能帮助后续 TileMap 拆解

概念图可采用宏观俯视或轻俯视视角。最终 Godot 实现仍以 RPG 斜顶式为准。

## Tile 标准

基础 tile 尺寸：

```text
32 x 32 px
```

第一版地图建议为中等规模。

参考尺寸：

```text
80 x 60 tiles
```

可根据实现调整，但不应过小。

## 地点列表

外部地图必须包含以下地点：

### 公共广场

功能定位：

- 小镇公共中心
- 公告、集会、公共事件的未来入口
- 地图空间中心之一

设计要求：

- 开阔
- 可走空间多
- 有公告板或中心标志物
- 周围道路连通多个区域

### 酒馆

功能定位：

- 未来社交、传闻、关系互动入口
- 不是完整室内玩法空间，只是外部入口地标

设计要求：

- 建筑体量适中，不要过大
- 门口清楚
- 可以有少量户外座位
- 与小巷相邻或接近

### 小巷

功能定位：

- 未来秘密、传闻、灰色事件、打听消息入口

设计要求：

- 必须真的像小巷
- 狭窄、阴影更重
- 位于建筑之间
- 有箱子、侧门、管道等元素
- 仍需保留可走通道，不能堵死

### 市场 / 商店街

功能定位：

- 未来购物、交易、摊位、资源流动入口

设计要求：

- 有摊位、遮棚、商品桌
- 中央行走通道要宽
- 不要让摊位堵住主路

### 政府大厅

功能定位：

- 未来公共事务、规则、职位、提案入口

设计要求：

- 有秩序感
- 有台阶、前庭、旗帜或公共建筑感
- 不要做成巨大城堡

### 档案馆

功能定位：

- 未来记忆、历史、身份、日报入口

设计要求：

- 安静、正式
- 有书馆/档案机构感
- 周围树木可以有，但不能太密

### 工坊

功能定位：

- 未来创造、制作、物件生成入口

设计要求：

- 有工作台、材料、烟囱、工具
- 材料堆靠边摆
- 中间留通道

### 未来扩展道路

功能定位：

- 表示小镇未来可扩展
- 后续可接入新区域或现实社区区域

设计要求：

- 地图边缘至少保留 1-2 条未开放道路
- 可以用路障、断桥、施工标识表示暂未开放

## 建筑尺寸原则

外部地图上的建筑是地点入口和地标，不是完整室内空间。

因此：

- 建筑不能过大
- 建筑不应占据过多可走面积
- 每个建筑必须有清楚门口
- 建筑之间要留出街道和广场
- 玩家未来移动时不应感到拥挤

## 障碍物原则

树、灯、摊位、箱子、装饰物都应减少阻塞。

原则：

- 装饰可以增加生活感
- 但不能妨碍移动
- 树木数量不要过多
- 主路和广场必须开阔
- 小巷可以窄，但不能难走

## 碰撞标准

Phase 0 定义规则，Phase 2 实现。

可走区域：

- 道路
- 广场
- 门口台阶
- 小巷通道
- 桥面
- 空地

不可走区域：

- 建筑主体
- 墙
- 树
- 水体
- 栅栏
- 大型装饰
- 摊位主体
- 路灯底座
- 工坊材料堆主体

碰撞判断原则：

> 基于现实逻辑判断用户是否能走过去。

## 地点入口标准

Phase 0-2 不做入口热区逻辑。

但必须预留清楚门口位置。

后续可以在这些门口添加：

- 靠近提示
- 进入按钮
- 场景切换
- 地点状态
- NPC 出现点

## 地点名称标准

地点名称不画死在地图素材里。

后续使用 Godot Label 显示。

原因：

- 方便改名
- 方便多语言
- 方便 UI 缩放
- 方便地图状态变化

## 昼夜标准

Phase 0-2 不实现昼夜系统，但地图必须支持未来昼夜扩展。

原则：

- tile 不做白天/夜晚两套完整版本
- 地图基础资产采用中性白天/偏傍晚光照
- 夜晚通过 Godot 环境遮罩、灯光、窗户发光实现
- 路灯位置需要在地图中预留
- 窗户可作为夜晚发光点

## 天气标准

Phase 0-2 不实现天气系统。

未来天气通过：

- 粒子
- 色调变化
- 雾层
- 雨层
- 少量地点状态变化

实现。

地图不应依赖天气专用资产才能成立。

## Phase 0-2 禁止事项

不要做：

- 玩家移动
- 摄像机
- NPC
- Agent runtime
- 室内地图
- 地点进入逻辑
- 天气系统
- 昼夜系统
- 完整经济
- 社交 UI

## 完成标准

Phase 0-2 完成后应具备：

- 地图标准明确
- 概念图已确认
- 外部地图场景可在 Godot 打开
- tile 结构合理
- 可走/不可走规则明确
- 碰撞已实际设置
- 后续移动引擎可接入测试
